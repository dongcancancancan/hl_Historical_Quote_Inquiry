# Routing Eval Stage Conclusion

## 一、阶段目标与已完成能力

当前阶段的开发目标已经完成，核心是验证 LLM 路由链路是否能稳定运行、沉淀真实样本，并辅助业务复盘，而不是接入正式金额计算。

已完成的能力包括：

- `route-test`：支持单条样本的 LLM 路由 dry-run。
- `route-runs`：支持查询历史路由日志，并按多种条件过滤。
- `review` 标注：支持业务对路由结果进行人工标注与备注。
- `summary/export`：支持汇总统计和样本导出。
- `full-price` 失败异步旁路日志：正式计算失败时可异步沉淀 `calculation_failure` 路由日志，不阻塞接口返回。
- 批量评估脚本：支持筛样本、批量执行 dry-run、导出结果 CSV。

## 二、Round1 / Round2 结论

### 1. Round1 结论

- Round1 已经把样本区分为“价格问题”和“路由问题”两类。
- 这一步很重要，因为价格缺失类样本不适合纳入路由准确率评估。
- Round1 的人工标注还暴露出一个关键问题：不少复杂样本并不是“单个 skill 选错”，而是天然存在“导体/编织阶段 + 胶料/后续制程阶段”的多阶段计算诉求。

### 2. Round2 结论

- Round2 在 `reason` 质量上有明显提升。
- 模型已经开始使用更贴近业务的表述，例如：
  - 多阶段计算可能性
  - `P` 结尾规格
  - 对绞配对关系
  - 顺序匹配规则
- 但 Round2 的核心结果没有发生变化：
  - `10/10` 样本仍然是 `manual_review`
  - `route_skill = 0`
  - `avg_confidence = 0.00`
- 这说明继续在“单一 `target_skill`”框架里调 prompt，收益已经非常有限。

### 3. 当前最重要的技术判断

- 现有 router 在单一 `target_skill` schema 下，已经基本触达了当前可解释能力上限。
- 它能更好地解释“为什么需要人工复核”，但还不能把复杂样本稳定收敛成一个可自动采用的单一 skill。
- 因此，下一阶段如果继续推进，不应再继续做 Round3 prompt 微调，而应转入 `route_plan` 多阶段路由设计。

## 三、当前 Router 的可用边界

当前 router 适合做的事情：

- 用于失败诊断和人工复核辅助。
- 用于指出“该样本可能需要多阶段计算”。
- 用于沉淀业务样本、业务标注和复盘证据。
- 用于把复杂失败样本的业务语言结构化，辅助后续设计。

当前 router 不适合做的事情：

- 不适合自动采用并直接驱动正式计算。
- 不适合直接作为金额计算入口。
- 不适合在复杂样本上直接输出单一可执行 skill。
- 不适合在未人工确认的情况下写入正式计算结果。

## 四、决策建议

基于当前阶段结果，建议如下：

- 暂不接自动采用。
- 暂不继续做 Round3 prompt 调优。
- 暂不修改正式 `full-price` 计算逻辑。
- 暂不修改当前 `decision_json` schema。
- 后续如果继续，应进入 `route_plan` 方案设计与评审，而不是继续单 skill prompt 调优。

## 五、下一阶段建议方向

下一阶段如继续推进，建议目标不是“让 LLM 直接选一个 skill 并自动计算”，而是：

- 让 LLM 输出多阶段路由建议。
- 每个阶段仍由确定性 skill 执行。
- 在人工确认前，不落正式计算结果。
- 继续保留 `manual_review` 作为低置信度和复杂场景的硬兜底。

一句话总结：

> 当前阶段的开发目标已经完成。下一步不是继续实现更多路由能力，而是沉淀结论，并等待业务确认是否值得进入多阶段 `route_plan` 路由开发。
