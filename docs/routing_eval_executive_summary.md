# Routing Eval Executive Summary

## 1. 我们验证了什么

本轮验证的目标不是让 LLM 直接参与金额计算，而是验证它是否适合做“成本分析计算前的路由判断”。

本次已经完成的验证范围包括：

- 路由策略配置与日志留痕
- `route-test` 单条 dry-run 验证
- `route-runs` 查询、标注、汇总、导出
- `full-price` 失败后的异步旁路日志
- 批量样本评估脚本
- Round1 / Round2 两轮真实样本复盘

一句话说，本轮已经完整验证了这条路线能不能作为“诊断与辅助决策工具”落地。

## 2. 结果怎么样

### 正向结果

- 这套路由链路已经可以稳定运行，并沉淀真实业务样本。
- LLM 的 `reason` 在 Round2 后明显更贴近业务语言，已经能表达：
  - 多阶段计算可能性
  - 对绞与 `P` 结尾规格的配对关系
  - 顺序匹配规则
- Round1 还帮助我们把“价格问题”和“路由问题”区分开了，避免误把缺价样本当成路由准确率问题。

### 关键限制

- Round2 的 `10` 条复杂样本，结果仍然是 `10/10 manual_review`
- `route_skill = 0`
- `avg_confidence = 0.00`

这说明：在当前“单一 `target_skill`”框架下，LLM 虽然更会解释，但还不能把复杂样本稳定收敛成一个可自动采用的单一 skill。

## 3. 为什么不能直接自动采用

原因不是链路没跑通，而是业务结构决定了当前 schema 不够用。

当前复杂样本普遍有这几个特点：

- 一张成本分析表同时包含导体/编织阶段和胶料/后续制程阶段
- 上下材料/制程数量不一致
- 存在对绞、集合、多铜绞、多芯押等复杂配对关系
- 一部分样本还夹杂铜价缺失、单价缺失等非路由问题

在这种情况下，如果仍要求 LLM 只能输出一个 `target_skill`，就会导致两种结果：

- 要么强行选一个 skill，风险很高
- 要么保守退回 `manual_review`

Round2 的结果已经证明，当前更接近真实业务的答案其实是“可能需要多阶段计算”，而不是“单一 skill 自动采用”。

## 4. 当前能上线用什么

当前最适合上线的定位不是自动计算，而是**内部诊断与人工复核辅助工具**。

可直接保留使用的能力：

- `route-test`：对单条复杂样本做 dry-run 判断
- `route-runs`：查看历史路由日志、原因和样本
- `review` 标注：业务可对结果标“对/错/备注”
- `summary/export`：支持复盘和样本导出
- `full-price` 失败异步旁路日志：正式计算失败时自动沉淀路由判断，不影响接口返回

推荐上线定位：

- 用于审价失败诊断
- 用于人工复核辅助
- 用于继续沉淀复杂制程样本

不建议当前上线为：

- 自动采用路由结果
- 自动驱动正式金额计算
- 直接替代现有确定性 skill

## 5. 后续如果继续，要投入什么

如果后续决定继续推进，就不再是小幅调 prompt 的问题，而是一个新的阶段性方案：

- `route_plan` 多阶段路由
- 人工确认机制
- step 级别材料/制程匹配
- step 级别校验与 trace
- 人工确认后再调用确定性 skill

这意味着后续工作量会明显上升，已经不适合在当前迭代里顺手继续做。

## 6. 推荐决策

### 短期建议

- 当前 LLM 路由适合作为审价失败诊断和人工复核辅助，不建议直接接入自动计算。
- 短期先把现有 `dry-run / route-runs / review / export` 作为内部分析工具保留。
- 继续收集样本，但不建议继续做 Round3 prompt 调优。

### 中期建议

- 复杂制程场景确实存在多阶段路由需求。
- 如果业务确认这条路线值得继续，应单独立项 `route_plan + 人工确认机制`。
- 在立项前，不建议再继续围绕“单一 `target_skill`”做更多调优。

## 最终结论

> 当前阶段已经完成验证闭环。  
> 结论不是“可以自动采用”，而是“这条路线适合作为诊断与辅助工具保留，复杂样本若要继续推进，需要进入下一阶段的 `route_plan` 多阶段路由设计与评审”。
